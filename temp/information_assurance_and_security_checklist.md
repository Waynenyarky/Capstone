# Information Assurance and Security Checklist

Each requirement has an ID (IAS-x.x), status, links to code/docs (relative from this file), a short description of how it was implemented, and how to test.

**Opening links:** Use Cursor’s “Open Link” (e.g. Ctrl/Cmd+Click) on the file path. Paths are relative to this file (`temp/`): `../backend/...` and `../docs/...` point to repo files.

### How to test IAS-3.3, 3.4, 3.6, 3.7 (step-by-step)

These are infrastructure/ops requirements; “testing” means **showing evidence** (screenshots, command output, or runbook text).

| ID | What to do | How to prove it |
|----|------------|-----------------|
| **3.3** (encryption at rest) | Proof is the **storage layer**, not something you see in the DB. (1) **Runbook:** *"Production database uses encryption at rest (Atlas default or encrypted host/volume)."* (2) **Show disk/volume encrypted:** Codespaces/Azure → cite “VM storage encrypted at rest” (provider docs); laptop → FileVault/BitLocker; Linux → `lsblk -f` or LUKS. See [security-database.md](../docs/security-database.md) §3.3. | **Evidence:** Runbook sentence and/or citation that the disk/VM where MongoDB runs is encrypted (e.g. Azure docs for Codespaces). |
| **3.4** (encrypted backups) | 1. In `.env` set `BACKUP_ENCRYPTION_PASSWORD=YourStrongPassword123!`<br>2. With stack running: `./deploy/backup.sh`<br>3. Check `backups/` — you should see a `.archive.enc` file (no plain `.archive`).<br>4. Restore test: `./deploy/restore.sh backups/capstone_backup_YYYYMMDD_HHMMSS.archive.enc` (with same password in `.env`) — should complete without error. | **Evidence:** (1) `ls backups/` showing `.archive.enc` and backup script output saying "Encrypting backup (AES-256-CBC)...", (2) Successful restore from `.archive.enc`. Without the password, `restore.sh` fails on `.enc` and `openssl` cannot decrypt — that proves encryption. |
| **3.6** (TLS DB connections) | **Production (Atlas):** Your `MONGO_URI` in production should start with `mongodb+srv://` (TLS by default). **Optional local demo:** Run `./scripts/mongo-tls-certs.sh` then `docker compose -f docker-compose.yml -f docker-compose.tls.yml up -d`; app services use a URI with `tls=true`. | **Evidence:** (1) Redacted screenshot or doc showing production URI uses `mongodb+srv://` or `tls=true`, or (2) Runbook sentence in [security-database.md](../docs/security-database.md) §3.6. For local TLS: after starting with TLS compose, auth-service logs should show "MongoDB connected" — connection uses TLS. |
| **3.7** (DB hardening) | 1. **Least privilege:** `docker compose up -d` then `docker compose exec mongodb mongosh -u capstone_app -p devapppass --authenticationDatabase admin --eval 'db.getSiblingDB("capstone_project").getCollectionNames()'` — should list collections. Try `db.getSiblingDB("admin").getCollectionNames()` as the app user — should fail (no admin access).<br>2. **Runbook:** [security-database.md](../docs/security-database.md) §3.7 has the hardening checklist (least privilege, network, password rotation, patching). | **Evidence:** (1) App user can read/write `capstone_project` but not admin DB (screenshot or copy-paste of commands). (2) Runbook sentence: *"Database hardening: app uses dedicated DB user with least privilege..."* — paste from security-database.md into your runbook. |
